# ZipLyne Lens redesign

**Date:** 2026-07-08
**Owner:** ZipLyne

## Goal
Rebuild the ZipLyne Lens desktop UI so it is clear, works end to end, and looks
like the provided design (`Tauri Desktop App Redesign` mockup). Non-technical
people should understand it instantly. The data engine (`@ziplyne/core`, API)
stays as-is; only the web UI (`apps/web`) is rebuilt.

## Terminology decision (locked)
The data has two levels: a billing entity matched from folder rules (old
"Client") and the git repos inside it (old "Project").

- Old **Client** -> renamed to **Project** everywhere in the UI (top-level,
  one clear total cost each).
- Old **Project** (`projectKey`, a git repo) -> renamed to **Repository**,
  shown nested inside each project's detail.
- Types in `@ziplyne/core` are NOT renamed; the mapping happens in the web layer.

## Constraints
- **No page scroll.** Every screen fits inside the window (min 1040x700, default
  1280x820). Internal panels may scroll only when genuinely necessary; pages do not.
- **Working navigation.** All 5 nav items render a distinct, real screen.
  (Today 4 of 6 nav items do nothing.)
- **No fabricated data.** Every number binds to the real summary/prompt library.
  Decorative-only metrics from the mockup (fake live burn, latency, vs-yesterday
  deltas with no basis) are replaced with real derivations or dropped.
- **Native window chrome.** Tauri uses the native title bar, so the app does not
  render a fake macOS traffic-light title bar.
- Match the mockup's visual language: light slate palette, blue (#2563eb) accent,
  soft cards, subtle gradients, tabular numerals.
- Non-technical friendly copy: plain language, no jargon, no em/en dashes in UI copy.

## Screens
1. **Overview** — greeting + total spend; KPI row (Spend, Tokens, Projects,
   Repositories) with real period deltas; spend-over-time area chart (from `days`);
   a highlight card (real cost velocity / top project); model mix + source mix bars;
   "Needs a project" unassigned queue; top projects table.
2. **Projects** — master list of projects (billable entities) with big clear cost;
   selected project detail: KPIs + repositories table (spend/tokens/last active per
   repo) + repo cost bars. This is the "clear what each project costs" centerpiece.
3. **Prompts** — library list + inspector, bound to the real prompt library.
4. **Models** — model cards (spend, tokens, cost per 1K) from real `models` rows.
5. **Settings** — connection/source status (real), config info (mode, scanned files,
   API url, rules), and local preference toggles (persisted to localStorage).

## Acceptance criteria (blind-verifier checks these)
1. Clicking each of the 5 nav items shows a visibly different, populated screen.
2. No screen scrolls the page vertically at 1280x820; content fits.
3. The word "Client" does not appear in the UI; the billable entity is "Project"
   and its git repos are "Repositories".
4. The Projects screen shows, for a selected project, its total cost and a per-repo
   cost breakdown.
5. Every displayed number traces to real API/summary data (no hardcoded mockup
   values like $128.47, 2,341 tok/min).
6. `pnpm --filter @ziplyne/web typecheck` passes; `pnpm typecheck` passes.
7. Works against both live local data and the demo fallback.
